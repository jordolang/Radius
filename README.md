# In The Mood ‚Äî full app

Anonymous, consent-first, proximity introductions for adults. Built so the
safety and privacy properties are enforced by the architecture, not just policy.

**Stack:** Next.js 15 (App Router) ¬∑ React 19 ¬∑ TypeScript (strict) ¬∑ Tailwind ¬∑
`sharp` for image de-identification ¬∑ designed for Vercel + managed Postgres.

Status: production build passes (18 routes), strict typecheck clean,
**103 unit tests passing** across the core logic.

## Run it
```bash
npm install
npm run dev        # http://localhost:3000
npm run build      # production build
npm run typecheck
```
## Database + auth setup

Persistence and identity are now real services. The in-memory `lib/store.ts`
and the `demoUserId` stub have been removed ‚Äî every route runs on Supabase +
the Clerk session.

- **Database/storage: Supabase Postgres.** Schema lives as migrations applied to
  the project; all access is server-side via the service-role client in
  `lib/db/*`. Every table is RLS-locked with no policies, so only the server
  reaches data. Identity is enforced from the session, never the request body.
- **Auth: Clerk.** `middleware.ts` + `<ClerkProvider>` wrap the app; the client
  reads identity via `lib/use-auth-user.ts` (`AuthGate`), and the server via
  `lib/auth.ts` (`getUserId` / `getUserIdEnsured`). No real name is ever stored;
  the Clerk user id is the foreign key across Supabase tables.

Configure `.env.local` (see `.env.example`):
```
NEXT_PUBLIC_SUPABASE_URL=‚Ä¶                 # provided
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=‚Ä¶     # provided
SUPABASE_SERVICE_ROLE_KEY=‚Ä¶                # Supabase dashboard ‚Üí Settings ‚Üí API (secret)
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=‚Ä¶        # create an app at dashboard.clerk.com
CLERK_SECRET_KEY=‚Ä¶
```
All routes run on Supabase + Clerk: profile, preferences, identity, presence,
alert, beacon, match, safety, plan, ephemeral, and signaling (signaling is a
DB-backed dev relay ‚Äî swap for a realtime provider at scale). Intimate-media
sharing (`ephemeral`) is gated on `users.age_verified`, which isn't wired to a
verification provider yet, so it stays off until that exists.

## Run on an iOS 26 simulator
The app ships a native iOS shell (Capacitor) so it runs as a real app on the
iOS Simulator. The shell is a WKWebView that loads the running Next.js server
at `http://localhost:3000` ‚Äî the iOS Simulator shares the macOS loopback, so
every API route and the in-memory store stay fully functional inside the app.

```bash
npm install
./scripts/run-ios.sh      # builds web + native, then installs & launches on a booted iOS 26 sim
```

Requirements: Xcode 26 with an **iOS 26** simulator runtime, Node 22+.

Notes on the native project (`ios/`, generated by `npx cap add ios`):
- **Programmatic root view controller.** `AppDelegate` creates the
  `CAPBridgeViewController` window in code instead of `Main.storyboard`, and the
  launch screen uses the Info.plist `UILaunchScreen` key. This avoids the
  Interface Builder compiler (`ibtool`/`actool`), which requires the simulator
  runtime to *exactly* match the SDK version ‚Äî a problem when the installed
  runtime (e.g. iOS 26.4) is older than Xcode's SDK (e.g. 26.5).
- **App Transport Security.** `Info.plist` allows local cleartext loads so the
  WKWebView can reach `http://localhost:3000` (development only).
- On first launch the WebKit GPU process can take a few seconds to warm up
  (a blank first paint); it renders immediately on subsequent launches.

## Map
The home screen hero is a live dark map centered on **you**. It renders with
**Mapbox GL JS** when `NEXT_PUBLIC_MAPBOX_TOKEN` is set, and falls back to free
**Leaflet + Carto** dark tiles otherwise, so the app still runs token-free.

Both engines honor the same privacy contract ‚Äî the map is intentionally *not* a
people-finder grid:
- A pulsing dot shows your device location, rendered **on-device only**; raw
  lat/lng is never transmitted (see `lib/geo.ts` / `lib/use-presence.ts`).
- A shaded rectangle is the **exact coarse ~5-mile cell** others can infer while
  you're available ‚Äî the honest representation of what's shared (`cellBounds`).
- A faint dashed ring is the proximity reach (how far a match can be).
- When `/api/alert` reports someone available nearby, an ambient halo animates on
  **your own** dot. No other user is ever plotted ‚Äî no pins, no count, no bearing.
  Connections happen through mutual opt-in (`/discover` ‚Üí interest ‚Üí `/match`).

Map components: `components/radius-map.tsx` (presence + control panel + engine
choice) delegating to `components/map/mapbox-canvas.tsx` or
`components/map/leaflet-canvas.tsx`.

**Pages** ‚Äî `/` availability + beacon ¬∑ `/onboarding` alias + profile + avatar ¬∑
`/discover` anonymous nearby ‚Üí interest ¬∑ `/plan` plan-ahead ¬∑ `/match/[id]`
consent gate ‚Üí chat ‚Üí pre-meet call.

**Libraries (all unit-tested)**
- `geo` coarse ~5mi cells; proximity from fuzzed centers; raw coords never stored
- `presence` available-now, auto-expiry, anonymous proximity alert
- `match` mutual-consent reveal ladder; real media needs second mutual consent
- `identity` pseudonyms from a pool; no real-name field exists
- `pii-guard` keeps emails/phones/handles/addresses out of chat
- `ephemeral` one-time, self-destructing media; gated by consent + age-verify
- `safety` private structured reports; graduated enforcement; severe fast-track
- `planahead` future availability; coarse regional-presence anti-abuse (not attendance)
- `beacon` real-time mutual proximity alerts; consent bright line; no crossed-paths log
- `profile` required descriptors + exact age/height/weight (filtered, shown only as coarse bands); private/optional health; non-deception attestation
- `preferences` each user's hard "absolutes" (age/height/weight ranges + gender); a **mutual** filter ‚Äî two people match only if each sits inside the other's ranges, else no candidate and no alert ever forms
- `avatarize` non-AI photo ‚Üí masked cartoon avatar; strips EXIF/GPS
- `premeet-signaling` live call state machine; never records

**Routes** under `app/api/*` wire those libraries; `app/api/signaling` is a dev
polling relay (swap for a realtime provider in production).

## Invariants held throughout
- Coarse location only; raw lat/lng never leaves the device or is stored.
- Matching is gated by mutual "absolutes": outside either person's age/height/weight
  ranges (or gender preference) ‚Üí no candidate, no alert, no visibility, even via direct URL.
- Pre-match attraction is photo-based but the photo is the **masked, de-identified
  avatar** + coarse bands; the real face / full profile unlocks only at mutual interest.
- Nothing else reveals between two users until BOTH opt in.
- No real names; no public health/disease field; no public partner "ratings".
- The pre-meet call is live and NEVER recorded ‚Äî only booleans + timestamps persist.
- Backing out of any meet is always free; penalties target abuse (scraping,
  plan-spam, regional no-shows), never declining.

## Known limits (honest)
- Screenshot prevention can be deterred/detected, never guaranteed (see ephemeral-viewer).
- Coarse-cell location can be spoofed; needs server-side anti-spoof signals later.
- Avatar masking deters but doesn't guarantee anonymity, especially combined with
  detailed attributes ‚Äî keep attributes coarse.
- Eye-region for the mask comes from a detector or manual placement (not built-in).
